# 项目治理

OpenClaw.NET 的治理目标是保持运行时对 .NET 开发者和运维人员有用、可自托管、安全且供应商中立。

治理决策应保护项目架构、贡献者信任以及下游在 OpenClaw.NET 之上构建商业或社区系统的自由。

## 原则

- 保持 OpenClaw.NET 默认可自托管、本地优先。
- 保持 NativeAOT 友好的核心路径。
- 保持可选集成明确、隔离且有文档记录。
- 偏好可复用基础设施而非特定产品的工作流逻辑。
- 为运行时、网关、安全、发布和扩展变更维护清晰的审查边界。
- 保持项目供应商中立，包括商业贡献者或赞助商参与时。
- 要求按项目许可证进行有意贡献。

## 角色

### 核心维护者

核心维护者负责影响项目长期健康的最终决策。其职责包括：

- 运行时架构
- 安全敏感变更
- 发布权限
- 包和项目边界
- 默认 Provider 行为
- 公开路线图方向
- 跨子系统边界变更的最终审查

影响运行时核心、网关安全态势、公开发布流程、包边界或默认运维行为的变更需要核心维护者批准。

### 区域维护者

区域维护者可以拥有限定范围的项目区域，例如：

- 文档
- 区域社区
- 行业扩展
- 示例
- 特定适配器
- 测试和兼容性文档

区域维护者可以在其范围内审查和指导 Pull Request。除非同时是核心维护者，否则他们对核心运行时架构、安全敏感变更、发布权限、包边界或项目范围的路线图决策没有最终决定权。

### 贡献者

贡献者通过代码、文档、测试、示例、Issue、审查、设计讨论和生态反馈来改进项目。

贡献应按项目许可证有意提交。贡献者应保持 Pull Request 聚焦、记录行为变更，并在贡献直接支持其公司或客户用例时进行披露。

### 赞助商

赞助商通过资金、基础设施、服务、文档帮助、示例、社区支持或其他资源支持项目。

赞助不授予：

- 路线图控制权
- 发布权
- 维护者身份
- 项目方向的独占所有权
- 对项目架构的优先控制权
- 供应商特定默认配置

赞助商可通过双方协议获得认可，但项目保持供应商中立。

## 维护者身份

维护者身份基于持续贡献、信任、技术判断、审查可靠性和社区行为。

维护者角色有范围限制。一个人可以在某个项目区域成为受信任的贡献者或区域维护者，而不对核心运行时、安全、发布或包边界决策拥有权限。

## 商业使用

欢迎将 OpenClaw.NET 用于商业用途。

公司和个人可以在 OpenClaw.NET 之上构建产品、集成、适配器、服务或内部平台。当贡献直接支持公司或客户用例时，贡献者应在 Issue、Pull Request 或提案中披露该背景。披露有助于审查者评估范围、中立性、安全态势以及该工作是否属于 core 还是扩展。

商业贡献者应避免将供应商特定默认配置、客户特定工作流或产品特定业务逻辑引入核心项目。

## 供应商中立性

OpenClaw.NET 必须保持供应商中立。

项目可以支持多种 Provider、适配器、协议和下游部署方式，但没有任何赞助商、公司或个人拥有生态方向。集成应保持用户选择权，避免将某个供应商或客户工作流设为默认路径。

## 行业包边界

行业包工作应保持为可复用基础设施、适配器、文档、示例和模板。

合适的行业包贡献包括：

- 协议抽象
- 遥测采集契约
- 适配器示例
- 模拟设备或机器示例
- 诊断和可观测性示例
- 部署指南
- 中英文文档

行业包贡献不应包括：

- 客户特定的工厂逻辑
- 专有数字员工工作流
- 闭源商业 UX
- 供应商专属默认配置
- 产品特定的业务规则

需要核心运行时行为、网关安全态势、包边界或发布流程更新的变更需要核心维护者审查。
